Phase 1
開発環境を整える
STEP 1-1
JDKをインストールする
⏱ 30分
›
🎯 ゴール：コマンドプロンプトで java -version を実行し、Java 17 が表示される状態にする。
手順
  1. Amazon Corretto 17 のダウンロードページを開く
  2. 「Windows x64 JDK」のインストーラー（.msi）をダウンロードする
  3. インストーラーを実行し、デフォルト設定でインストール（「JAVA_HOME を設定する」にチェックが入っていることを確認）
  4. コマンドプロンプトを新規で開き、以下を実行する
cmd
java -version
javac -version
✅ 成功の確認

openjdk version "17.x.x" 2023-xx-xx
OpenJDK Runtime Environment Corretto-17.x.x
javac 17.x.x

⚠️ よくあるつまずき

「'java' は内部コマンドまたは外部コマンドとして認識されていません」と出た場合は、コマンドプロンプトを再起動してください。それでも出る場合は環境変数 JAVA_HOME を手動設定します。

STEP 4-3
テーブルを作成する（DDL）
⏱ 45分

🎯 ゴール：DB設計書のテーブルを Oracle に作成し、JPA の Entity クラスと紐づけられた状態にする。
まず USERS テーブルから作成する
DDL（SQL）
CREATE TABLE users (
    id          NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        VARCHAR2(100)  NOT NULL,
    email       VARCHAR2(255)  NOT NULL UNIQUE,
    password_hash VARCHAR2(255) NOT NULL,
    role        VARCHAR2(20)   DEFAULT 'USER' NOT NULL,
    address     CLOB,
    created_at  TIMESTAMP      DEFAULT SYSTIMESTAMP NOT NULL,
    updated_at  TIMESTAMP      DEFAULT SYSTIMESTAMP NOT NULL
);
User.java（Entity）
@Entity
@Table(name = "users")
@Data
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(name = "password_hash", nullable = false)
    private String passwordHash;

    @Enumerated(EnumType.STRING)
    private Role role = Role.USER;
}

STEP 5-4
注文確定APIを実装する（トランザクション処理）
⏱ 2〜3日
›
🎯 ゴール：注文確定時に「在庫減算」「注文レコード作成」「決済処理」が一つのトランザクションで動く状態にする。
実装の流れ
  1. カート内容を取得する
  2. 在庫の悲観ロック（SELECT FOR UPDATE）で在庫数を確保する
  3. orders レコードを作成する
  4. order_items レコードを作成する
  5. 決済モック処理を実行する
  6. 決済成功 → 在庫を減算、注文ステータスを CONFIRMED に更新する
  7. 決済失敗 → 例外を throw してロールバックさせる
OrderService.java（骨格）
@Service
public class OrderService {

    @Transactional
    public OrderResponse placeOrder(Long userId) {
        // 1. カート取得
        Cart cart = cartRepository.findByUserId(userId)
            .orElseThrow(() -> new CartNotFoundException());

        // 2. 在庫確認・悲観ロック
        cart.getItems().forEach(item -> {
            Stock stock = stockRepository
                .findByProductIdWithLock(item.getProduct().getId())
                .orElseThrow();
            if (stock.getQuantity() < item.getQuantity())
                throw new InsufficientStockException();
        });

        // 3-4. 注文・明細作成、5-7. 決済→在庫減算 or ロールバック
        // ... 実装を続ける
    }
}
🔰 確認問題

「悲観ロック」と「楽観ロック」の違いを説明してください。ECサイトの在庫更新にはどちらが向いていますか？理由も答えましょう。

STEP 5-5
共通エラーハンドラを実装する
⏱ 1日
›
🎯 ゴール：全APIで統一されたエラーレスポンス形式が返る状態にする。
GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult().getFieldErrors()
            .stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.toList());
        return new ErrorResponse("VALIDATION_ERROR", errors);
    }

    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleGeneral(Exception ex) {
        return new ErrorResponse("SERVER_ERROR", ex.getMessage());
    }
}

STEP 6-3
注文確定の異常系テストを書く
⏱ 0.5日

テストすべき異常系ケース

on:
  push:
    branches: [ develop, feature/** ]
  pull_request:
    branches: [ develop, main ]

  test:
    runs-on: ubuntu-latest

      - uses: actions/checkout@v4

      - name: Set up Java 17

STEP 7-3
CD：SSH で Ubuntu Server に自動デプロイする
⏱ 1日
›

SSH鍵の準備
  1. Ubuntu Server 上で SSH キーペアを生成する
  2. 公開鍵を ~/.ssh/authorized_keys に追加する
  3. 秘密鍵を GitHub Secrets の SSH_PRIVATE_KEY に登録する
.github/workflows/cd.yml（デプロイステップ追加）
      - name: Deploy to Ubuntu Server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/ec-site
            sudo systemctl stop ec-site-api || true
            sudo cp ~/ec-site-api.jar /opt/ec-site/app.jar
            sudo systemctl start ec-site-api
            sleep 5
            systemctl is-active ec-site-api

STEP 8-2
systemd サービスとして登録する
⏱ 30分
›
🎯 ゴール：アプリが systemd サービスとして起動し、サーバー再起動後も自動起動する状態にする。
/etc/systemd/system/ec-site-api.service
[Unit]
Description=EC Site API
After=network.target

[Service]
User=ubuntu
WorkingDirectory=/opt/ec-site
ExecStart=/usr/bin/java -jar /opt/ec-site/app.jar
EnvironmentFile=/opt/ec-site/.env
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
bash
sudo systemctl daemon-reload
sudo systemctl enable ec-site-api
sudo systemctl start ec-site-api
sudo systemctl status ec-site-api
