EC バックエンド API 開発ロードマップ（図解付き・学習用）

本書は、Windows 11 上で Spring Boot（Java / Amazon Corretto 21）を用いて EC バックエンド API を開発し、
VirtualBox 上の Ubuntu Server（SSH ポート 8022）へ GitHub Actions で CI/CD デプロイするまでの流れを、
「OS ユーザー分離」「DB ユーザー分離」「リバースプロキシ（Nginx/Apache）」を含めて体系的に学ぶための技術書スタイルのドキュメントです。

読み方のガイド：
第 1 章〜第 3 章で環境とユーザー設計を理解し、
第 4 章〜第 6 章でアプリとインフラの連携、
第 7 章〜第 8 章で CI/CD と運用の全体像を掴む構成になっています。

第 2 章　OS ユーザー設計

2.1 OS ユーザー分離の目的

  • アプリが侵害されても、サーバー全体への被害を最小化する
  • デプロイ操作とアプリ実行を分離し、責務を明確にする
  • ログやファイルの所有者を分けて、運用・監査をしやすくする
図 2-1　OS ユーザー構成図
+------------------+      +------------------+      +------------------+
|  admin (初期)    |      |  deploy          |      |  ecapp           |
|  sudo 権限あり   |      |  デプロイ専用    |      |  アプリ実行専用  |
+---------+--------+      +---------+--------+      +---------+--------+
          |                       |                       |
          | 管理作業              | SSH 8022 で接続       | systemd からのみ実行
          v                       v                       v
   OS 設定変更・                  デプロイ用ディレクトリ  Spring Boot アプリ
   パッケージ導入                 への jar 配置           のプロセス実行

2.2 ユーザー作成手順

2.2.1 デプロイ用ユーザー（deploy）

sudo adduser deploy
sudo usermod -aG sudo deploy

2.2.2 アプリ実行専用ユーザー（ecapp）

sudo adduser ecapp --disabled-password --shell /usr/sbin/nologin
sudo mkdir -p /home/ecapp/app
sudo chown -R ecapp:ecapp /home/ecapp/app
ポイント：
ecapp ユーザーは SSH ログイン不可にし、systemd からのみアプリを実行させることで、
「アプリ実行権限」と「サーバー操作権限」を明確に分離します。

第 3 章　DB ユーザー設計

3.1 DB ユーザー分離の目的

  • アプリからは最小限の権限（SELECT/INSERT/UPDATE/DELETE）のみを付与する
  • DB 管理者（postgres）とアプリユーザーを分けて、誤操作や侵害時の被害を抑える
図 3-1　DB ユーザー権限図
+------------------+          +------------------+
| postgres         |          | ecapp_user       |
| DB 管理者        |          | アプリ専用       |
+---------+--------+          +---------+--------+
          |                             |
          | CREATE DATABASE / 管理      | SELECT / INSERT / UPDATE / DELETE
          v                             v
                 +----------------------+
                 |   ecapp_db           |
                 +----------------------+

Spring Boot アプリ
        |
        v
  ecapp_user 経由で ecapp_db に接続

3.2 DB ユーザー作成手順

sudo -u postgres psql

-- アプリ専用ユーザー
CREATE USER ecapp_user WITH PASSWORD '強いパスワード';

-- アプリ用データベース
CREATE DATABASE ecapp_db OWNER ecapp_user;

-- 公開権限を削除し、アプリユーザーにのみ接続を許可
REVOKE ALL ON DATABASE ecapp_db FROM PUBLIC;
GRANT CONNECT ON DATABASE ecapp_db TO ecapp_user;
学習ポイント：
アプリからは決して postgres ユーザーで接続しないこと。
これは「root でアプリを動かさない」のと同じくらい重要な原則です。

4.2 Nginx を使う目的

  • HTTPS（TLS）の終端を担当し、アプリは HTTP のままでよい
  • 外部公開ポート（80/443）を一箇所に集約し、セキュリティを高める
  • 静的ファイルを高速に配信できる
  • 将来的に複数アプリへの負荷分散も可能

4.3 Nginx 設定例

sudo apt install -y nginx

sudo nano /etc/nginx/sites-available/ecapp

server {
    listen 80;
    server_name _;

    location / {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

sudo ln -s /etc/nginx/sites-available/ecapp /etc/nginx/sites-enabled/ecapp
sudo nginx -t
sudo systemctl restart nginx

4.4 Apache での代替

Apache でも mod_proxy を使えば同様のリバースプロキシ構成が可能です。
ただし、設定ファイルやモジュール構成がやや複雑になりがちで、学習コストを考えると Nginx から入る方が理解しやすいケースが多いです。

第 6 章　Spring Boot アプリ設計

6.1 レイヤ構成

com.example.ecapp
├─ Application.java
├─ config        # セキュリティ・Swagger・DB 設定
├─ controller    # REST API エンドポイント
├─ service       # ビジネスロジック
├─ repository    # DB アクセス (Spring Data JPA)
├─ entity        # JPA エンティティ
├─ dto           # リクエスト/レスポンス DTO
└─ exception     # 例外・エラーハンドリング

6.2 重要な設計ポイント

  • JWT 認証（Authorization: Bearer <token>）
  • BCrypt によるパスワードハッシュ
  • @Transactional による注文確定＋在庫更新＋決済モックの一括処理
  • 決済失敗時は例外を投げてロールバック
  • @ControllerAdvice による統一エラーレスポンス

第 7 章　systemd による常駐化

7.1 サービスファイル

sudo nano /etc/systemd/system/ecapp.service

[Unit]
Description=EC App
After=network.target

[Service]
User=ecapp
WorkingDirectory=/home/ecapp/app
ExecStart=/usr/bin/java -jar app.jar --spring.profiles.active=prod
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target

sudo systemctl daemon-reload
sudo systemctl enable ecapp
sudo systemctl start ecapp
sudo systemctl status ecapp
学習ポイント：
systemd によって、サーバー再起動後も自動的にアプリが起動し、
異常終了時には自動再起動されるようになります。

第 8 章　CI/CD（GitHub Actions）

8.1 CI/CD フロー図

図 8-1　CI/CD シーケンス図
開発者          GitHub Actions          Ubuntu Server          systemd
  |                   |                       |                    |
  |  git push         |                       |                    |
  |------------------>|                       |                    |
  |                   |  ビルド & テスト      |                    |
  |                   |---------------------->|                    |
  |                   |                       |                    |
  |                   |  SSH (8022, deploy)   |                    |
  |                   |---------------------->|                    |
  |                   |                       |  app.jar 更新      |
  |                   |                       |------------------->|
  |                   |                       |                    |  再起動
  |                   |                       |                    |

8.2 Secrets 設定

  • SERVER_HOST：Ubuntu Server の IP アドレス
  • SERVER_USER：deploy
  • SERVER_SSH_KEY：deploy ユーザーの秘密鍵
  • SERVER_SSH_PORT：8022

8.3 ワークフロー例

name: CI-CD

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up JDK 21 (Corretto)
        uses: actions/setup-java@v4
        with:
          distribution: 'corretto'
          java-version: '21'
      - name: Build with Maven
        run: mvn -B clean package

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Copy jar & restart service
        uses: appleboy/ssh-action@v0.1.7
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          port: ${{ secrets.SERVER_SSH_PORT }}
          script: |
            set -e
            cd /home/deploy
            # ここでは GitHub から scp で jar を送る構成などに合わせて調整
            # 例: /home/ecapp/app/app.jar を更新
            sudo systemctl restart ecapp

第 9 章　学習のためのまとめ

  • OS ユーザー分離（admin / deploy / ecapp）で責務と権限を分ける
  • DB ユーザー分離（postgres / ecapp_user）で最小権限を徹底する
  • Corretto 21・Maven・IntelliJ を揃えて、開発と本番の差異を減らす
  • Nginx/Apache をリバースプロキシとして使い、HTTPS と公開ポートを一元管理する
  • systemd でアプリを常駐化し、再起動や障害時の復旧を自動化する
  • GitHub Actions でビルド〜デプロイを自動化し、変更を安全かつ素早く反映する
