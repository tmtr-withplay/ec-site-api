1 準備フェーズ（詳細手順）

1.1 Windows 11 開発機：Amazon Corretto 21 の導入

開発・本番で同一の JDK を使うことが重要です。Corretto 21 をインストールして IntelliJ と Maven が参照する JDK に設定します。

# 手順（概略）
1. Amazon Corretto 21 のインストーラ（Windows MSI）をダウンロード
   - Corretto の公式ダウンロードページから "Amazon Corretto 21" を選択
2. MSI を実行してインストール
3. 環境変数を確認（PATH に java が通る）
   > java -version
   # 出力例: openjdk version "21" 2024-09-xx ...
4. IntelliJ の設定
   - File > Project Structure > SDKs で Corretto 21 を追加
   - Project SDK と Module SDK を Corretto 21 に設定
ポイント：JDK のマイナーバージョン差で動作が変わることがあるため、開発機とサーバーで同じ Corretto 21 のビルドを使ってください。

1.2 Maven を手動でインストールし IntelliJ で利用する手順

IntelliJ は内蔵の Maven を使えますが、CI と同じローカル Maven を使うことで差異を減らします。以下は Windows での手順です。

# 1. Maven のダウンロードとインストール（Windows）
1. https://maven.apache.org/download.cgi から Binary zip をダウンロード
2. zip を展開（例: C:\tools\apache-maven-3.9.x）
3. 環境変数を設定
   - M2_HOME=C:\tools\apache-maven-3.9.x
   - MAVEN_HOME=C:\tools\apache-maven-3.9.x
   - PATH に %M2_HOME%\bin を追加
4. 動作確認:
   > mvn -v
   # Java version: 21 (Corretto 21) が表示されること

# 2. IntelliJ で外部 Maven を使う設定
1. File > Settings > Build, Execution, Deployment > Build Tools > Maven
2. "Maven home directory" を "Bundled (Maven x.x.x)" から "External" に切り替え、C:\tools\apache-maven-3.9.x を指定
3. "User settings file" や "Local repository" を必要に応じて指定（CI と同じ settings.xml を使う場合はコピー）
4. プロジェクトを再インポート（右上の Maven アイコン > Reimport All Maven Projects）
理由：CI（GitHub Actions）とローカルで同じ Maven バージョン・settings.xml を使うと、依存解決やビルド結果の差異を減らせます。

1.3 VirtualBox 上の Ubuntu Server（GUI なし）準備（要点）

  • Ubuntu Server をインストール
  • SSH のポートを 8022 に変更済み（サーバー側の sshd_config で Port 8022）
  • ホストからはサーバーの IPアドレス を使って接続（例: ssh -p 8022 deploy@192.168.56.101）
  • サーバーに Corretto 21、PostgreSQL、Nginx（または Apache）をインストール
# SSH 接続例（ホスト側）
ssh -i ~/.ssh/deploy_key -p 8022 deploy@192.168.56.101
重要：VirtualBox のネットワーク設定は「ブリッジ」または「ホストオンリー」など、IP アドレスでアクセスできる方式にしてください。NAT のままではホストから直接 IP でアクセスしづらくなります。

1.4 Ubuntu 側：Corretto 21 / PostgreSQL / Nginx の導入（要点）

# Ubuntu の例（概略）
sudo apt update && sudo apt upgrade -y

# Corretto 21 (deb パッケージを使用する例)
wget https://corretto.aws/downloads/latest/amazon-corretto-21-x64-linux-jdk.deb
sudo dpkg -i amazon-corretto-21-*-linux-x64.deb
java -version

# PostgreSQL
sudo apt install -y postgresql postgresql-contrib

# Nginx (リバースプロキシ)
sudo apt install -y nginx
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
# SSH ポート 8022 を許可
sudo ufw allow 8022/tcp

2 DB クライアント：A5 Mk-2 SQL の利用

A5 Mk-2 SQL を使う理由：Windows 環境での軽量な SQL クライアントで、PostgreSQL に接続可能。接続情報（IP, port, DB 名, ユーザー, パスワード）を設定して利用します。

# A5 Mk-2 SQL の接続設定（概略）
- Host: 192.168.56.101
- Port: 5432
- Database: ec_db
- User: ec_user
- Password: (設定したパスワード)
A5 Mk-2 は Windows ネイティブの GUI クライアントです。pgAdmin や DBeaver と比べて軽量で操作が速い点が特徴です。

3 Nginx の目的と Apache での代替検討

3.1 Nginx をリバースプロキシに使う目的（初心者向け）

  • TLS 終端（HTTPS 化）：Nginx が外部からの HTTPS 接続を受け、内部の Spring Boot アプリ（通常は localhost:8080）へ HTTP で転送します。これによりアプリ本体は TLS を意識せずに済みます。
  • 静的コンテンツ配信：画像や CSS、JS などを Nginx が高速に配信できます（アプリは API のみ担当）。
  • 負荷分散・プロキシ機能：将来的に複数インスタンスを立てる場合、Nginx がロードバランサとして振る舞えます。
  • セキュリティの分離：公開用のポート（80/443）は Nginx が受け、アプリは内部ポートで動かすことで攻撃面を限定できます。

3.2 Apache で代替できるか（結論と比較）

結論：はい、Apache でも代替可能です。ただし用途や運用方針で選択が変わります。

観点NginxApache
パフォーマンス（静的配信）高い（イベント駆動で少ないメモリ）良好（モジュールによるがプロセスモデルでメモリ増加しやすい）
リバースプロキシ機能広く使われる（設定が簡潔）可能（mod_proxy を使用）
設定の簡潔さ比較的シンプルモジュールが多く柔軟だが設定が複雑になりがち
運用実績コンテナ/クラウド環境で広く採用従来の LAMP 系で長年の実績あり

運用上の推奨：初心者で「軽くて設定が分かりやすい」「高い同時接続性能」を重視するなら Nginx を推奨します。既に Apache に慣れている、または Apache 固有のモジュール（.htaccess など）を使いたい場合は Apache を選んでも問題ありません。

4 CI/CD（GitHub Actions）とデプロイ（SSH 8022 / IP アドレス）

4.1 ワークフローのポイント（更新）

  • ビルドは GitHub Actions 上で Corretto 21 を使う（actions/setup-java の distribution: 'corretto', java-version: '21'）。
  • デプロイは SSH ポート 8022 を使ってサーバーの IP アドレスへ接続する。GitHub Secrets に SERVER_HOST（IP アドレス）、SERVER_SSH_PORT=8022、SERVER_USER、SERVER_SSH_KEY を登録する。
  • デプロイ後は systemd でアプリを管理し、Nginx（または Apache）で 80/443 を受けて内部のアプリへプロキシする。
# GitHub Actions の deploy スニペット（要点）
- name: Set up JDK 21 (Corretto)
  uses: actions/setup-java@v4
  with:
    distribution: 'corretto'
    java-version: '21'

- name: Deploy to server via SSH
  uses: appleboy/ssh-action@v0.1.7
  with:
    host: ${{ secrets.SERVER_HOST }}        # IP アドレスを指定
    username: ${{ secrets.SERVER_USER }}
    key: ${{ secrets.SERVER_SSH_KEY }}
    port: ${{ secrets.SERVER_SSH_PORT }}    # 8022
    script: |
      # デプロイ処理（scp 等）と systemd 再起動
GitHub Actions からサーバーへ接続する際は、サーバー側で公開鍵を authorized_keys に登録し、秘密鍵を GitHub Secrets に登録してください。秘密鍵にはパスフレーズを設定することを推奨します。

5 実装・運用上の注意（Corretto 21 / Maven / Nginx/Apache）

  • 依存性の固定：pom.xml の依存バージョンは明示的に固定し、CI でキャッシュを使って再現性を高める。
  • JVM オプション：本番ではメモリ設定（-Xms/-Xmx）を systemd の ExecStart に指定して安定稼働させる。
  • プロファイル：Spring のプロファイル（dev/prod）を使い、DB 接続やログレベルを切り替える。
  • プロキシ設定：Spring Boot を Nginx/Apache の背後で動かす場合、X-Forwarded-* ヘッダを正しく扱う設定を行う（Spring の ForwardedHeaderFilter 等）。
